## Repo-specific setup

- This repo is an app (Vite site), not a publishable library — no `dist/` entry, no `.d.ts`. The build runs in **synth-entry mode**: `package-build.mjs` is invoked with `--entry ./dist/index.js` (a path that deliberately doesn't exist) purely so the script walks up from its dirname to find this repo's real `package.json` (for `PKG_DIR`/naming) and then falls back to scanning `src/components/ui/` directly. This is expected and prints `[NO_DIST]` — not an error to chase.
- `cfg.srcDir` is scoped to `src/components/ui/` (not the repo's `src/`) so the converter only picks up the 3 actual design-system primitives (Button, Badge, Eyebrow) and not layout/page components (Nav, Footer, HomePage, PaletteSamplePage).
- **`cfg.cssEntry`/`cfg.tokensGlob` point at the compiled Vite output** (`dist/assets/index-<hash>.css`), not `src/index.css`. The raw source uses Tailwind v4's `@theme {}` at-rule and `@import "tailwindcss"`, which are Tailwind-build-time-only syntax — a browser (and the claude.ai/design renderer) ignores the unrecognized `@theme` at-rule entirely, so none of the `--apd-*` custom properties would actually be defined if the raw file were shipped. **Before every re-sync**: run `npm run build` (`cfg.buildCmd`) fresh, then `ls dist/assets/*.css` and update `cssEntry`/`tokensGlob` in config.json to the new hashed filename — Vite content-hashes it on every build, so this path is not stable across builds.
- No Storybook, no `docs/`/`guides/` — `.prompt.md` files are synthesized from `.d.ts` + JSDoc + the authored preview compositions, not ported docs.

## Component scope decisions

- Only `Button`, `Badge`, `Eyebrow` exist in this repo's `src/components/ui/`. The claude.ai/design project ("APD Design System") previously had additional components (`Logo`, `Stat`, `Callout`, `Card`, `Input`) grouped under `brand/buttons/feedback/forms/surfaces` — these had no corresponding source anywhere in this repo (checked via grep across `src/`), so they were almost certainly authored directly in the Claude Design tool, not from a prior design-sync run. They were deleted at upload time per the "no anchor, review before delete" atomic-path rule, with the user's explicit sign-off before starting (re-adopt vs. fresh-project choice).
- The remote project's pre-existing structure (`<group>.card.html` cards, `SKILL.md`, `_adherence.oxlintrc.json`, `ui_kits/`, `Canvas.dc.html`, `uploads/`) does not match anything design-sync itself produces — it's very likely the project was built up through the Claude Design app directly (or an older/different tool), not a prior design-sync run. Those files are outside design-sync's declared write/delete globs (components/**, tokens/**, fonts/**, _vendor/**, _preview/**, guidelines/**) so they were left untouched.

## Re-sync risks

- **`cssEntry` will go stale on every rebuild** unless the hashed filename is refreshed first (see above) — this is the single most likely thing to silently break a future sync.
- If real Card/Input/Callout-equivalent primitives are ever added to `src/components/ui/`, they'll be picked up automatically (srcDir scope covers the whole `ui/` folder).
- Preview compositions in `.design-sync/previews/*.tsx` were ported from `HomePage.tsx`'s real usage (hero CTAs, "Not a broker" badge, section eyebrows) as of 2026-07-03 — if that page's compositions change meaningfully, the previews won't auto-follow; re-author by hand if they drift far from real usage.
- No fonts were bundled (`[FONT_REMOTE]` — Archivo/Inter/Merriweather/Montserrat load via a Google Fonts `@import` in the compiled CSS, which ships as-is and loads at runtime in the design tool). Nothing to fix here unless the DS ever needs to work offline.
- `guidelinesGlob`/`docsDir` are both unset (no `docs/`/`guides/` in this repo) — if brand guideline pages (color/spacing/type reference cards) are wanted again, they'd need to be authored as `docs/guides/*.md` in-repo, or added back by hand in the Claude Design app (outside design-sync's scope either way).
