## Repo-specific setup

- This repo is an app (Vite site), not a publishable library — no `dist/` entry, no `.d.ts`. The build runs in **synth-entry mode**: `package-build.mjs` is invoked with `--entry ./dist/index.js` (a path that deliberately doesn't exist) purely so the script walks up from its dirname to find this repo's real `package.json` (for `PKG_DIR`/naming) and then falls back to scanning `src/components/ui/` directly. This is expected and prints `[NO_DIST]` — not an error to chase.
- `cfg.srcDir` is scoped to `src/components/ui/` (not the repo's `src/`) so the converter only picks up actual design-system primitives, not layout/page components (Nav, Footer, HomePage, CtaBand — CtaBand lives in `src/components/layout/`, out of scope). As of 2026-07-14 that's 9 components: `Button`, `Badge`, `Eyebrow`, `ContactForm`, `PageHero`, `SectionHead`, `ServiceCard`, `StatsRow`, `TextLink`. `Container` also lives in `src/components/ui/` but is deliberately excluded via `componentSrcMap: {"Container": null}` — it's a plain max-width/padding div wrapper with no visual identity of its own, not a brand/visual component worth a DS card.
- **`TextLink` and `ServiceCard` need a router context** — both render react-router's `<Link>` internally and throw `Cannot destructure property 'basename' of ... useContext(...) as it is null` if mounted bare. Fixed via `cfg.provider: {"component": "MemoryRouter"}` + `cfg.extraEntries: ["react-router-dom"]` (react-router-dom was already being inlined as a bundled dep since `TextLink.tsx` imports from it — extraEntries just re-exposes `MemoryRouter` on `window.APDUI` so the provider wrapper can reference it). This is documented in `conventions.md` since the design agent needs to know to wrap compositions using either component in `MemoryRouter`.
- **Adding `cfg.provider` changes the render hash of EVERY component**, not just the ones that need it — a resync after adding it cleared previously-`good` grades for `Badge`/`Button`/`Eyebrow` even though nothing about them changed (the provider wraps every preview). Re-graded from fresh sheets, visually identical to before (the router wrapper renders no visible chrome) — expected, not a regression to chase.
- **Full-resolution source photography breaks the 12MB per-file upload cap.** `ServiceCard`'s and `PageHero`'s authored previews import real product photos from `src/assets/` for realism — the raw files are multi-MB (`pexels-*.jpg` etc. run 2-4MB each), and esbuild inlines an imported image as a base64 data URI in synth-entry mode (no url-loader configured), so `_preview/ServiceCard.js` hit 13.9MB and failed `[FILE_TOO_LARGE]`. Fixed by generating small preview-only copies via `sips -Z 480 -s formatOptions 60` into `.design-sync/previews/assets/` (tens-to-low-hundreds of KB each) and importing those instead of the real site assets. **If more previews need real photography, resize into `previews/assets/` the same way rather than importing `src/assets/*` directly.**
- `StatsRow`'s authored previews show count-up numbers mid-animation in the graded screenshot (e.g. "+70%" instead of the final "+80%") — the component's count-up is scroll-triggered via `IntersectionObserver` + 1s `requestAnimationFrame` easing with a 300ms stagger per stat, and the capture script screenshots before it settles. Graded `good` anyway: this is a capture-timing artifact, not a defect — a live-rendered card (an iframe actually mounting the component, as the design tool does) runs the same animation to completion within ~1.3s of mounting, same as the real site.
- `[GRID_OVERFLOW]` fired on `StatsRow` (4-column stat rows are wider than a standard grid cell) — fixed with `cfg.overrides.StatsRow: {"cardMode": "column"}`, applied via a targeted `preview-rebuild.mjs --components StatsRow` (no full re-validate needed — column cards can't re-flag `wide`).
- **`cfg.cssEntry`/`cfg.tokensGlob` point at the compiled Vite output** (`dist/assets/index-<hash>.css`), not `src/index.css`. The raw source uses Tailwind v4's `@theme {}` at-rule and `@import "tailwindcss"`, which are Tailwind-build-time-only syntax — a browser (and the claude.ai/design renderer) ignores the unrecognized `@theme` at-rule entirely, so none of the `--apd-*` custom properties would actually be defined if the raw file were shipped. **Before every re-sync**: run `npm run build` (`cfg.buildCmd`) fresh, then `ls dist/assets/*.css` and update `cssEntry`/`tokensGlob` in config.json to the new hashed filename — Vite content-hashes it on every build, so this path is not stable across builds.
- No Storybook, no `docs/`/`guides/` — `.prompt.md` files are synthesized from `.d.ts` + JSDoc + the authored preview compositions, not ported docs.

## Component scope decisions

- Only `Button`, `Badge`, `Eyebrow` exist in this repo's `src/components/ui/`. The claude.ai/design project ("APD Design System") previously had additional components (`Logo`, `Stat`, `Callout`, `Card`, `Input`) grouped under `brand/buttons/feedback/forms/surfaces` — these had no corresponding source anywhere in this repo (checked via grep across `src/`), so they were almost certainly authored directly in the Claude Design tool, not from a prior design-sync run. They were deleted at upload time per the "no anchor, review before delete" atomic-path rule, with the user's explicit sign-off before starting (re-adopt vs. fresh-project choice).
- The remote project's pre-existing structure (`<group>.card.html` cards, `SKILL.md`, `_adherence.oxlintrc.json`, `ui_kits/`, `Canvas.dc.html`, `uploads/`) does not match anything design-sync itself produces — it's very likely the project was built up through the Claude Design app directly (or an older/different tool), not a prior design-sync run. Those files are outside design-sync's declared write/delete globs (components/**, tokens/**, fonts/**, _vendor/**, _preview/**, guidelines/**) so they were left untouched.

## Re-sync risks

- **`cssEntry` will go stale on every rebuild** unless the hashed filename is refreshed first (see above) — this is the single most likely thing to silently break a future sync.
- If real Card/Input/Callout-equivalent primitives are ever added to `src/components/ui/`, they'll be picked up automatically (srcDir scope covers the whole `ui/` folder).
- Preview compositions in `.design-sync/previews/*.tsx` were ported from `HomePage.tsx`'s real usage (hero CTAs, "Not a broker" badge, section eyebrows) as of 2026-07-03 — if that page's compositions change meaningfully, the previews won't auto-follow; re-author by hand if they drift far from real usage.
- No fonts were bundled (`[FONT_REMOTE]` — Archivo/Inter/Merriweather/Montserrat load via a Google Fonts `@import` in the compiled CSS, which ships as-is and loads at runtime in the design tool). Nothing to fix here unless the DS ever needs to work offline.
- `guidelinesGlob`/`docsDir` are both unset (no `docs/`/`guides/` in this repo) — if brand guideline pages (color/spacing/type reference cards) are wanted again, they'd need to be authored as `docs/guides/*.md` in-repo, or added back by hand in the Claude Design app (outside design-sync's scope either way).
- `.design-sync/previews/assets/*.jpg` (small resized copies for `PageHero`/`ServiceCard` previews) are a NEW durable asset this run added — commit them alongside the previews that reference them, or those two previews break on a fresh clone.
- If `src/components/ui/` grows a new component that also needs `MemoryRouter` (anything importing `Link`/`NavLink`/`useNavigate` etc.), no config change needed — the existing `cfg.provider` already covers it.
